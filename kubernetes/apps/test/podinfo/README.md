# Podinfo

Podinfo is a deliberately small web application commonly used to demonstrate
and test Kubernetes delivery systems. It exposes request and runtime information
and provides predictable health endpoints, making it useful for proving that a
cluster path works without introducing application-specific infrastructure.

Here Podinfo is not meant to represent the final application architecture. It
is the final checkpoint for Cilium, Flux, Helm, app-template, Gateway API, Envoy,
Services, and pod networking.

## app-template

The workload uses the bjw-s-labs `app-template` Helm chart rather than a
Podinfo-specific chart. app-template converts a structured values document into
ordinary Kubernetes controllers, Services, storage objects, and routes. This
provides a consistent chart interface that can later be reused for many
self-hosted applications.

The current Helm values create:

- one controller containing `ghcr.io/stefanprodan/podinfo:6.14.0`;
- one container listening on port `9898`;
- one Service exposing that port;
- one HTTPRoute attached to the `network/envoy` Gateway's `http` listener.

The route has no hostname or path match, so it accepts any HTTP request reaching
the listener. This is intentional for the first test: no DNS entry or `Host`
header is required.

## Files

```text
podinfo/
├── app/
│   ├── ocirepository.yaml   # app-template chart, currently 5.0.1
│   ├── helmrelease.yaml     # Podinfo image, Service, and route values
│   └── kustomization.yaml
└── ks.yaml                  # waits for the Gateway before deployment
```

The HTTPRoute is generated by Helm, so it is not visible as a standalone YAML
file in Git. Inspecting the rendered cluster resources is part of understanding
what app-template produced.

## Verify the workload

```sh
flux -n test get helmrelease podinfo
kubectl -n test get deployment,pod,service,httproute
kubectl -n test describe httproute podinfo
kubectl -n test logs deployment/podinfo
curl http://192.168.1.120
```

If the pod and Service are healthy but the request fails, inspect the HTTPRoute
status. Its parent conditions explain whether the route was accepted by the
Gateway and whether its backend reference resolved successfully.
